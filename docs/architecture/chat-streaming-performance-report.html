Chat Streaming Performance Deepening Report

Focus: the two user-visible outliers in Normal Chat streaming: the pre-stream "Preparing response" wait and the post-stream interval where the answer has stopped changing but the UI still shows generation in progress.

stream-completion.ts

Docs Check

  • SvelteKit 2: route adapters may return a ReadableStream; cancellation is request-signal driven and headers must describe the stream.
  • Svelte 5: UI side effects should sit in $effect cleanup paths; touched files should keep modern event attributes.
  • AI SDK 6: fullStream exposes text, reasoning, tool, usage, and finish events; finish metadata does not need to block visible token delivery.

ADR Constraints

  • ADR-0015: durable Normal Chat turn completion stays in chat-turn; route adapters must stay thin.
  • ADR-0019: browser turn transitions stay in normal-chat-client-turn-runtime.ts, above stream transport.
  • ADR-0025/0026: AI SDK UI stream framing and Vercel AI SDK model run are current implementation facts; do not revive the retired custom SSE grammar.

The current implementation has useful modules, but two interfaces are shallow for latency: the route adapter returns the stream only after preflight, and terminal stream frames are emitted only after broad durable completion plus projection work.

1. Fast Terminal Receipt Module

highest leverage post-stream lag

Files involved

  • src/lib/server/services/chat-turn/stream-completion.ts:332
  • src/lib/server/services/chat-turn/finalize.ts:430
  • src/lib/client/normal-chat-client-turn-runtime.ts:448
  • src/lib/services/streaming.ts:615

What leaks

The browser-facing terminal interface leaks the full completion implementation. The UI cannot receive data-stream-metadata, finish, or [DONE] until message writes, turn state, context sources, compression snapshots, cost totals, and generated-output reconciliation are done.

Dependency category

Hidden coupling between durable completion, read projection, generated file reconciliation, and browser generation state. 

flowchart TB
  subgraph Before
    A["upstream finish"] --> B["finalizeChatTurn"]
    B --> C["context sources"]
    C --> D["cost and snapshots"]
    D --> E["file reconciliation"]
    E --> F["terminal frames"]
  end
  subgraph After
    G["upstream finish"] --> H["persist receipt"]
    H --> I["terminal frames"]
    H --> J["background projections"]
    J --> K["conversation detail refresh"]
  end

Problem

sendEndAndClose is defined inside completeStreamTurn and emits terminal frames after optional fallback context-source construction, compression snapshot listing, and cost lookup. Its callers first await finalizeChatTurn, whose implementation creates messages, persists turn state, starts evidence, builds completion context sources, and may reconcile generated outputs.

Solution

Deepen stream-completion.ts around a small terminal receipt interface: persist the minimal user and assistant identities, token counts, model/depth metadata, stopped/incomplete flags, and generation duration, then emit terminal frames immediately. Move broad projections behind an asynchronous follow-up that updates the existing Conversation Detail Read Model and evidence/file polling paths.

Wins

  • Clears generation state as soon as the answer is durably identifiable.
  • Keeps ADR-0015 intact because completion still belongs to chat-turn.
  • Improves locality: terminal emission has one implementation and one focused test surface.
  • Preserves existing sidecar paths: evidence polling, file job refresh, and conversation hydration.

ADR callout: compatible with ADR-0015 and ADR-0022 if read projection remains in Conversation Detail and route adapters stay thin.

2. Stream Admission Before Heavy Turn Preparation

high leverage pre-stream wait

Files involved

  • src/routes/api/chat/stream/+server.ts:32
  • src/lib/server/services/chat-turn/preflight.ts:28
  • src/lib/server/services/chat-turn/stream-orchestrator.ts:659
  • src/lib/server/services/chat-turn/stream-orchestrator.ts:1111

What leaks

The route adapter interface makes the client wait for work that could be reported as stream activity. Because the stream is not returned until after preflight, the prelude and "context preparing" activity cannot cover conversation lookup, attachment readiness, linked-source writes, skill resolution, and depth metadata.

Dependency category

Leaky adapter sequencing: transport setup is coupled to turn preparation instead of cheap admission. 

flowchart TB
  subgraph Before
    A["POST"] --> B["parse"]
    B --> C["capacity"]
    C --> D["preflight"]
    D --> E["return stream"]
    E --> F["prelude and activity"]
  end
  subgraph After
    G["POST"] --> H["parse and admission"]
    H --> I["return stream"]
    I --> J["prelude"]
    J --> K["preflight activity"]
    K --> L["model prep and stream"]
  end

Problem

The route records route parse, capacity, and preflight timings, but those marks are only exposed through headers/terminal metadata. When preflight is slow, users still see the generic placeholder with no live server-side activity.

Solution

Split preflightChatTurn into a cheap admission module and a stream-time preparation module. Admission keeps HTTP status errors for malformed input and capacity. Later preparation errors are emitted as AI SDK UI stream error frames. The orchestrator can then send the prelude and visible response activity before heavy work starts.

Wins

  • Reduces time to response headers and first byte.
  • Makes "Preparing context" visible during real preparation work.
  • Keeps route adapters thin by moving preparation into chat-turn.
  • Creates a precise seam for timing and failure tests around admission vs preparation.

ADR callout: compatible with ADR-0019 and ADR-0025 if the emitted errors remain AI SDK UI stream parts consumed by streaming.ts.

3. Context Preparation Stage Module With Timing Leverage

diagnostic leverage pre-stream wait

Files involved

  • src/lib/server/services/normal-chat-context-preparation.ts:4
  • src/lib/server/services/chat-turn/streaming-normal-chat-model-run.ts:159
  • src/lib/components/chat/MessageBubble.svelte:384
  • src/lib/services/stream-timeline.ts:1

What leaks

The implementation has a real stage plan, but the public activity interface collapses all stage detail into one "context preparing" row. That hides which stage dominates latency and prevents targeted reductions such as budgeting forced web prefetch or automatic compression separately.

Dependency category

Shallow telemetry interface over a deeper staged implementation. 

flowchart TB
  subgraph Before
    A["plan"] --> B["context"]
    B --> C["compression"]
    C --> D["web prefetch"]
    D --> E["prompt budget"]
    A -. "one generic activity" .-> F["Preparing context"]
  end
  subgraph After
    G["stage runner"] --> H["sanitized activity"]
    G --> I["stage timings"]
    I --> J["slow-stage budgets"]
  end

Problem

The stage runner has depth internally, but callers do not get enough leverage from its interface. The browser sees only a generic label, and logs get terminal timings too late to explain an active slow turn.

Solution

Deepen the stage module so it returns sanitized stage timings and emits localized, non-diagnostic activity classes. Keep raw stage ids out of visible copy, but make them available to stream timing and focused tests. Then add per-stage budgets where the outliers prove stable.

Wins

  • Separates visibility from implementation detail.
  • Improves locality for context-preparation performance work.
  • Turns future latency fixes into stage-local changes.
  • Supports focused regression tests without adding route-local logging.

ADR callout: compatible with the context-selection seam if ranking and prompt assembly remain in their existing modules.

4. Runtime Phase Model: Generating vs Finalizing

UI leverage post-stream clarity

Files involved

  • src/lib/client/normal-chat-client-turn-runtime.ts:259
  • src/lib/client/normal-chat-client-turn-runtime.ts:448
  • src/lib/components/chat/MessageBubble.svelte:389
  • src/lib/components/chat/MessageInput.svelte:1833

What leaks

The runtime exposes a binary sending state. The visual modules must infer "generation" from isStreaming and isThinkingStreaming, so post-token persistence appears as continued generation.

Dependency category

Duplicated phase meaning between stream transport, runtime state, composer controls, and message rendering. 

flowchart TB
  subgraph Before
    A["active stream"] --> B["isSending true"]
    B --> C["composer shows stop"]
    B --> D["message shows generating"]
  end
  subgraph After
    E["active stream"] --> F["preparing"]
    F --> G["streaming"]
    G --> H["finalizing"]
    H --> I["done"]
  end

Problem

Even after terminal emission is made faster, the runtime would benefit from a phase interface instead of several booleans. The current booleans are enough to operate, but shallow for user perception and queue behavior.

Solution

Add a small phase enum to the runtime snapshot and adapt visual modules to use that phase for label and control decisions. The phase should be driven by decoded stream callbacks, not by route-local state.

Wins

  • Stops presenting persistence wait as model generation.
  • Gives the composer a cleaner interface for stop, queue, and disabled states.
  • Improves locality: one runtime owns phase semantics.
  • Pairs well with candidate #1 but can be tested independently.

ADR callout: compatible with ADR-0019 because browser-side turn transitions remain in the runtime module.

Recommended Order

  1. Candidate #1 first: it attacks the largest post-stream wait without changing model behavior.
  2. Candidate #2 second: it shortens or at least makes visible the pre-stream wait, but changes error timing semantics.
  3. Candidate #3 third: use it to measure and budget the remaining pre-stream work.
  4. Candidate #4 alongside #1 or after it: it turns latency wins into clearer user perception.
