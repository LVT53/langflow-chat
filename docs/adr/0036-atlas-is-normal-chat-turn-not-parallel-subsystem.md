# Atlas is a Normal Chat Turn + artifact, not a parallel background subsystem

The previous Deep Research feature was a massive parallel background subsystem — 16 database tables, 7 configurable LLM roles, a 60-second-tick worker, a duplicated model runner, and a human-in-the-loop plan approval gate. It bypassed Normal Chat Turn Completion entirely, never integrated with streaming, and "literally never worked reliably and always showed bullshit." We are replacing it with **Atlas**: a durable report artifact produced by a special kind of Normal Chat Turn. The kickoff turn completes through `finalizeChatTurn` with explicit memory/Honcho skips, while the long-running background job runs an enforced multi-stage research pipeline through the existing `normal-chat-model`, SearXNG search, and file-production infrastructure. There are 2 Atlas-owned tables (not 16), no duplicated model runner, no plan approval gate, and no parallel subsystem.

## Considered Options

1. **Atlas as a Normal Chat Turn + artifact (chosen)** — the turn creates a job, the kickoff assistant message completes through `finalizeChatTurn`, and the job runs the 7-stage pipeline in the background. The completed artifact is stored via file-production and linked back to the Atlas job/message. The server orchestrates a fixed stage sequence (decompose → search → curate → synthesize → integrate → assemble → audit); the model fills in content within each stage. Quality gates replace plan approval. 2 Atlas-owned tables. Reuses existing chat, model, search, generated-document, and read-model infrastructure.

2. **Fixed background worker subsystem (rejected)** — keep the old architecture but fix the 10 identified failure modes (wire streaming, use `normal-chat-model`, integrate `finalizeChatTurn`, make it a tool). Rejected because the subsystem complexity itself was a root cause of failure: 16 tables, 7 model roles, 15 env vars, plan-approval state machine, duplicated runner. Fixing all of that while keeping the shape would still leave a maintenance burden and integration surface that a Normal Chat Turn does not have.

3. **Atlas as a Skill Session (rejected)** — a persistent skill that changes model behavior for research-heavy conversations, with the report emitted via `produce_file` at session end. Rejected because skills shape prompts and session behavior, not multi-step tool loops with enforced stage sequences. The skill system is about instruction injection, not pipeline orchestration.

## Consequences

- Atlas cannot run as a fully synchronous streamed turn for long research (10-30 minutes). It creates a background job instead, which means the user can leave the page. This is a deliberate trade: we lose in-stream token streaming but gain fire-and-forget UX and avoid holding a stream slot for 20 minutes.
- The server-orchestrated fixed pipeline means the model cannot deviate from the stage sequence. This is the feature, not a limitation — model deviation (rabbit-holing, skipping curation, early stopping) is exactly what produced "bullshit" in the old pipeline and in naive deep research agents.
- Quality gates can extend the pipeline beyond the profile's target goals. Even the quickest profile (Overview) cannot ship if the audit finds unverified claims. This means a "quick" Atlas might take longer than expected — but it will never ship gibberish.
- The anti-hallucination architecture uses a different model family for the citation audit than for synthesis. Same-model audit was theatre in the old pipeline. This adds one model configuration requirement.
- No plan approval gate means the user cannot edit sub-queries before research starts. They can cancel and restart with a more specific query if the direction is wrong. The decompose stage's sub-queries are visible in the progress indicator for transparency.
- The code-execution harness approach (Parallel's architecture, where the model writes Python that calls search primitives) was evaluated and noted as a future evolution. It was not chosen for v1 because it works best with models trained for code-based research orchestration; with general-purpose LLMs, a fixed server-orchestrated pipeline is more reliable and testable.

## Resolved Grilling Branches

The following branches were resolved through grilling and recorded in CONTEXT.md and/or this ADR.

1. ~~Failure, cancellation, and resume~~ — **RESOLVED**: cancellation discards partial state, stale running jobs recover via heartbeat timeout (10min) like file-production, and failed jobs retain the latest checkpoint for inspection/version seeding. V1 retries rerun the bounded pipeline from the durable job state rather than replaying every intermediate stage.
2. ~~Cost & limits~~ — **RESOLVED**: per-Atlas usage tracking accumulates token usage from all model calls and stores it on `atlas_jobs`; the cost field is stored with the job and remains zero when the provider/model runtime does not expose pricing. User-facing usage/cost is shown after completion in Atlas job details. 1 active Atlas per user, global admin-configurable limit (default 2), no per-day/month quotas in v1.
3. ~~Concurrency~~ — **RESOLVED**: queue, don't reject. Per-user limit (1) enforced at creation time. Global limit enforced at claim time — job stays queued until a slot frees. User sees "waiting for a research slot" in progress indicator. No estimated wait time in v1.
4. ~~Which models~~ — **RESOLVED**: 2 configs — ATLAS_SYNTHESIS_MODEL (decompose/curate/synthesize/integrate/assemble, defaults to user's selected chat model) + ATLAS_AUDIT_MODEL (citation audit, defaults to the "other" model for cross-model verification). Both admin-configurable via config-store with provider:<id>:<modelId> format. If only one model enabled, audit falls back to it with warning. No 7-role config.
5. ~~SearXNG dependency~~ — **RESOLVED**: Atlas requires SearXNG. If not configured, Atlas button/command is disabled with a localized explanation. No knowledge-only fallback in v1 — without web search, it's not Atlas. Regular Normal Chat with knowledge context already exists for that use case.
6. ~~Knowledge library integration~~ — **RESOLVED**: Atlas Local Sources come from explicit linked sources/composer attachments plus readable active working-set documents selected through the existing knowledge/context boundary. Explicit user-selected sources are snapshotted to the kickoff user message and receive highest authority; automatic working-set sources have lower authority. Report distinguishes "Web Sources" and "Your Library" in Sources section. Memory is background context, not a citable source.
7. ~~Privacy & data lifecycle~~ — **RESOLVED**: Atlas jobs follow the same privacy lifecycle as file-production jobs. Clear Workspace Data: cancel active + delete all atlas jobs/state/files. Clear Memory & Knowledge: doesn't touch Atlas (workspace, not memory). Account Erasure: cancel + delete everything. Account Data Archive: includes produced Atlas files as generated files, NOT research state. Conversation deletion: cancel + delete job rows. Wired into existing privacy-controls, no separate Atlas privacy branch.
8. ~~Completion notification~~ — **RESOLVED**: three layers — on-page progress card polls + updates in real-time, conversation list sidebar badge on focus/visibility refresh, browser push notification via Web Push API (VAPID) when user has left AlfyAI. Push notifications are new infrastructure (no service worker or web-push dep exists today) — Atlas is the first consumer but the push subscription system is app-level, not Atlas-specific.
9. ~~Report structure~~ — **RESOLVED**: freeform with mandatory sections (Title, Executive Summary, freeform body, Limitations, Sources). No intent classification, no fixed templates. Model decides headings/structure/order based on content. Sidebar navigation auto-generated from heading blocks. Old pipeline's intent-specific shapes were a classification failure point — removed.
10. ~~Language~~ — **RESOLVED**: decompose detects user language via existing language.ts. All stage prompts in detected language. Report in detected language. Search queries language-agnostic (search where sources are, write in user's language). Source citations keep original language titles. HTML UI chrome localized via i18n. Hungarian Parity Check in audit stage to catch English slippage.
11. ~~Graceful "not enough evidence" outcome~~ — **RESOLVED**: one outcome — Atlas always ships. If evidence is thin, Limitations section is prominent and honest. Executive Summary states the limitation upfront. Body contains what was found with honesty markers. Quality gates block gibberish/fabrication, NOT honest gaps. No separate "memo" or "limited" outcome — the honesty markers + Limitations section already communicate evidence quality.
12. ~~HTML renderer JS~~ — **RESOLVED**: Atlas emits a deterministic `GeneratedDocumentSource` and delegates HTML/PDF/Markdown projection to the existing standard report renderer in file-production. Renderer CSS/JS remain app-owned and deterministic; the model never writes presentation code. Model's job is content, source structure, and honesty markers, not presentation.

## Resolved Grilling Branches (round 2)

13. ~~The `create_atlas` contradiction~~ — **RESOLVED**: server creates the Atlas job directly when it detects `atlasMode: true` in the send payload. No model tool call. The "compiling" message is a canned template (query echoed, profile described, "you can close this page"). The model's intelligence is spent in the background pipeline stages, not the kickoff turn. The `create_atlas` function is a server-side call in the send route handler, not a model-facing AI SDK tool — same pattern as how `/web` sets `forceWebSearch: true`.
14. ~~Conversation sealing~~ — **RESOLVED**: Atlas does NOT seal conversations. After Atlas completes, the user can continue chatting, ask follow-ups, start another Atlas, or use Continue/Fork/Revise. The sealedAt column stays in schema (vestigial) but Atlas never sets it. CONTEXT.md flagged ambiguities about "Report Boundary" and "finished chat" are historical Deep Research concepts.
15. ~~`atlasMode` + profile flow~~ — **RESOLVED**: 2 new fields on ChatTurnRequest: `atlasMode` (boolean, default false) + `atlasProfile` (enum: overview/in-depth/exhaustive, null when not atlas). When `atlasMode` is true, send route short-circuits: creates job, writes canned message, calls finalizeChatTurn — skips context-selection, model-run, deliberation entirely. Existing reasoningDepth, forceWebSearch, and pendingSkill are ignored when atlasMode is true. Linked sources and attachments are not prompt context for the kickoff turn; they are handed to Atlas as explicit local-source inputs per branch 18. Zero impact on existing turns (atlasMode defaults false). Minimal surface: 2 fields, 1 early-return in preflight, 1 job-creation path in send route.
16. ~~Atlas + depth system~~ — **RESOLVED**: Atlas pipeline bypasses the depth system entirely. Kickoff has no model call (canned message). Background pipeline stages use normal-chat-model programmatic API with hand-crafted prompts, not the depth/deliberation system. Atlas Profile (Overview/In-Depth/Exhaustive) controls pipeline parameters (source/round targets, quality gate strictness), not the Normal Chat depth profile.
17. ~~Skills + Atlas interaction~~ — **RESOLVED**: skills are silently ignored during Atlas Turns. Atlas pipeline uses hand-crafted stage-specific prompts — injecting skill instructions would pollute the structured pipeline. Skill chip stays visible but pendingSkill is not passed to the Atlas job (per branch 15). No error or blocking — just ignore. Cleaner than old pipeline's explicit `if (turn.deepResearchDepth) return null` check.
18. ~~Linked sources and attachments~~ — **RESOLVED**: linked sources (/source) and composer attachments are snapshotted on the kickoff user message and included as Atlas Local Sources with the HIGHEST authority boost — above automatic working-set documents and parent-seed sources. Passed to decompose stage as "user-provided sources" before web search. Used to inform sub-query generation. Shown in report Sources under "Your Library" with a "You provided these" indicator.
19. ~~Atlas versioning semantics~~ — **RESOLVED**: `parent_atlas_id` is the immediate parent (A→B→C chain). New Atlas: parent=null, new document family. Continue: parent=previous, same family, seeded with compressed research state + curated sources. Revise: parent=previous, same family, seeded with GeneratedDocumentSource (structure awareness) + fresh searches. Fork: parent=previous, new family, seeded with compressed research state (optional). Uses existing artifact/document-family version history. Continue = "go deeper," Revise = "update with new info," Fork = "new trajectory from parent."
20. ~~SearXNG rate limiting~~ — **RESOLVED**: 3-layer protection. (1) Batched search with configurable concurrency (default 3 parallel, `ATLAS_SEARCH_CONCURRENCY`). (2) Inter-batch delay (default 500ms, `ATLAS_SEARCH_BATCH_DELAY_MS`). (3) Graceful degradation on 429/timeout: exponential backoff (500ms→10s max), if >50% batch fails → stop searching + proceed with what we have, Limitations section notes "Search was rate-limited." Quality gate checks if enough sources were gathered for the profile.
21. ~~Atlas worker startup~~ — **RESOLVED**: separate `ensureAtlasWorker()` in hooks.server.ts init, same pattern as `ensureFileProductionWorker()`. Separate worker from file-production — claims `atlas_jobs`, not `file_production_jobs`. Reuses the worker-runner pattern (claim→execute→store→complete) but has its own claim query, execution function, heartbeat. Admin-configurable enable/disable via config-store, defaulting to enabled.
22. ~~Atlas completion → memory feedback~~ — **RESOLVED**: the kickoff turn uses Normal Chat Turn Completion and normal message analytics/evidence metadata, but Atlas background completion does not create a second chat turn, `atlas_completed` analytics event, task-state note, Honcho enrichment, or automatic memory intake. Completion records profile/source/usage/output state on `atlas_jobs`, links the file-production job and generated files to the kickoff assistant message, and may send an app-level browser push notice. Atlas content follows Assistant Prose Memory Exclusion — it's assistant-generated, not user truth. If user wants to remember something from the Atlas, they use existing memory intake in a follow-up chat.
23. ~~Atlas UI design~~ — **RESOLVED**: a live HTML prototype was built at `docs/prototypes/atlas-prototype.html` demonstrating all 9 UI surfaces. The user reviewed it across 3 iterations and approved the overall approach with refinements, including dark-mode expectations. The prototype's visual style (warm off-white, terracotta accent, Nimbus Sans L UI / Libre Baskerville body, quiet reading-first spacing) should be the reference for implementation — the final Atlas UI should match this prototype's look and feel.

**Design decisions locked from prototype review:**
- Report viewer style: mix of A (refined current — left-border accent headings, generous whitespace) and C (modern documentation — sans-serif body font, sidebar active-section border design). No drop-caps, no numbered section prefixes, no magazine layout.
- Source chips: small favicon-only circles (no border, no background color), hover tooltip shows favicon + page title on line 1, compact LLM reasoning on line 2. No full-width source links anywhere.
- Honesty markers: colored pills (green/yellow/red/orange) with icon + text vertically centered. Hover shows verdict + audit reasoning.
- Completion card: "Open" is the only text button; Download/Continue/Fork/Revise are icon-only ghost buttons with hover tooltips. No limitation notice (it's in the report). Profile shown as just the name, not "Profile: In-Depth".
- Kickoff message: same card style as progress card, with an animated SVG of Earth + a plane orbiting it. Smooth fade-out transition when SVG switches from kickoff animation → progress ring → completion checkmark.
- Progress card: just "ATLAS" label + ring animation + cycling status messages + Cancel. No profile text, no extra dot.
- Sidebar badge: simple colored circle (no icon), coexists with existing hover 3-dots menu icon.
- Full EN/HU language switching via toggle — all UI chrome switches, report content stays in its research language.
- Continue/Fork/Revise open inline expansion panels below the completion card, not modals.
- Images in reports with caption + underlined accent-colored source attribution links.
- Spacing: gap between last Web Sources item and "Your Library" subtitle.
- Dark mode: all Atlas surfaces, including the report viewer, must support the app's existing dark mode using the existing dark-mode token system from `app.css` and auto-detection via `prefers-color-scheme` / `.dark` class.

## Locked Implementation Edge Cases

These are implementation-level decisions, not new product-domain branches.

1. **Kickoff transport**: Atlas creation is a send-route operation, not a stream-route operation. The client should route `atlasMode: true` submissions to `/api/chat/send`, receive the canned kickoff assistant message and initial Atlas job/progress-card state as JSON, then refresh/poll conversation detail for progress. `/api/chat/stream` must not create Atlas jobs; if `atlasMode` reaches stream by mistake, it should return a clear client error instead of falling back to a normal streamed chat turn. This preserves the fire-and-forget design and avoids holding stream slots for Atlas work.

2. **Normal Chat Turn Completion without memory intake**: Atlas kickoff uses the shared finalization boundary for conversation persistence, assistant metadata, normal message analytics/evidence metadata, and generated-file/job link readiness, with explicit options that skip raw assistant-prose memory intake and Honcho enrichment. Background completion then updates the Atlas job with usage, source counts, file-production job id, generated-file ids, and optional browser push. The canned kickoff message and generated Atlas content are assistant-authored status/output, not user truth.

3. **Idempotency**: Atlas job creation must be idempotent across double-clicks, retries, reloads, and reconnect-like client recovery. The client sends and persists a stable Atlas turn id with the draft/request, and the server enforces a unique job key scoped to user, conversation, Atlas action (`create`, `continue`, `revise`, `fork`), parent Atlas id when present, profile, normalized query hash, and that client turn id. A repeated request with the same key returns the existing linked job/message state rather than creating a second job; a cancelled or failed unlinked placeholder can be replaced on retry.

4. **Checkpoint state granularity**: V1 persists a completed Atlas checkpoint after audit and before deterministic rendering, not after every individual stage. The checkpoint contains the curated source pool with source authority labels, compressed findings, accumulated usage/cost, quality-gate diagnostics, and the produced GeneratedDocumentSource summary. Cancellation discards partial state and never resumes.

5. **Linked source and attachment readiness**: Explicit linked sources and composer attachments are highest-authority Atlas Local Sources only after they are openable and readable through the existing working-document/knowledge boundaries. Atlas creation waits for normal attachment intake readiness within the existing preflight timeout. The resolved explicit source set is snapshotted to the kickoff user message so later link changes cannot alter a running job's local source pool. If an explicit source cannot become readable, the kickoff fails with a localized source-readiness error rather than silently dropping user-provided material. Automatic working-set sources and parent-seed sources may degrade gracefully; explicit user-selected sources may not.

6. **Always ships vs quality gates**: Quality gates may request one audit-driven revision pass, but they must not loop indefinitely. After that bounded revision is exhausted, Atlas still ships an honest report if rendering/storage can succeed: the Executive Summary and Limitations section must state the evidence gap, and unsupported areas must carry honesty markers. Only infrastructure failures that prevent creating a trustworthy artifact at all, such as renderer/storage failure or no readable explicit required source, leave the Atlas job failed.

7. **Primary output identity**: The styled HTML Atlas is the primary generated document and opens by default in Document Workspace. PDF and Markdown are sibling downloadable outputs attached to the same file-production job and document family. Continue and Revise operate from the persisted GeneratedDocumentSource/document-family identity, not from the PDF or Markdown projection. Fork creates a new document family seeded from the parent Atlas state.

8. **Push notification scope**: Web Push is optional app-level notification infrastructure. Missing VAPID keys, denied notification permission, unsupported browser APIs, or service-worker registration failure disable only the browser push layer. On-page progress polling and the conversation-list sidebar badge still work. Push subscription storage, permission UI, and service-worker code must be app-owned and reusable, with Atlas as the first producer, not Atlas-specific notification plumbing.
