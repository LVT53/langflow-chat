# Memory Rework Update keeps Honcho-led memory usable long-term

AlfyAI memory will remain Honcho-led and background by default, but it will no longer be treated as fully autonomous in every case. When memory maintenance can confidently deduplicate, expire active use, or recognize a clear supersession, it should act without asking the user. When remembered facts are ambiguous, contradictory, stale in a user-impacting way, sensitive, or otherwise require user authority, AlfyAI should create a Guided Memory Review item in the Memory Profile.

This is a complete Memory Rework Update, not a provisional v1, draft, or temporary cleanup pass. The end state should be coherent enough to run unattended while AlfyAI accrues real Memory Rework Telemetry about intake quality, maintenance behavior, review burden, and prompt-use outcomes. Implementation may still land in safe changesets, but the product design should be end-to-end before the rework is considered complete.

This decision extends, but does not replace, the Honcho-led memory decision in `0011-honcho-led-memory-context.md`. Honcho remains the authority for persona and relationship memory. The app-owned Memory Profile Projection is the user-facing read model, not a parallel persona-memory system. Guided Memory Review exists because the user is the correct authority for some memory decisions, especially when the system cannot safely infer which remembered fact should remain active.

Honcho should not be used as a raw transcript mirror. Conversation history remains available through the app's chat history and retrieval paths, but ordinary user and assistant messages should not automatically become durable persona memory. New material should pass through a Memory Intake Gate before it reaches durable memory: explicit durable facts, preferences, constraints, goals, and concrete work capsules may be admitted, while ordinary task chatter, assistant prose, and obvious junk should not be mirrored just because they occurred. The gate should make a bounded structured Memory Intake Decision before any durable memory write; it should not persist a separate long-lived pending-memory-candidate store that can drift into a second memory system.

A Memory Intake Decision has three outcomes: admit, reject, or defer to maintenance. Admit writes accepted durable memory through the existing memory authority path. Reject records the reason the material should not become durable memory. Defer to maintenance is the uncertainty path: it records privacy-preserving telemetry and dirty state so later bounded maintenance can inspect the account, but it does not store raw candidate text or create a durable maybe-memory backlog.

Immediate Memory Admission should have a higher threshold than general memory detection. Preferences, constraints, goals, project rules, and document rules should require explicit durable language or strong explicit phrasing before being admitted immediately. Stable user-authored self-statements are the exception: users naturally say "I live in Amsterdam" or "my company is X" without saying "remember this." Those may be admitted when they are clearly about the user, first-party, and not merely document-derived or assistant-inferred. Ambiguous self-statements, soft instructions, and inferred traits should defer to maintenance or review rather than become immediate durable memory.

Assistant prose should be excluded from Immediate Memory Admission. Ordinary assistant-generated answer text may remain chat history, evidence, or conversation context, but it should not become durable memory authority. Assistant prose can contain guesses, summaries, caveats, examples, document interpretations, or repeated noisy memory; mirroring it into durable memory creates a feedback loop where model-generated claims become more "confirmed" because the model said them. App-owned structured outputs, such as work capsules, task continuity updates, generated document metadata, skill notes, or typed tool outcomes, may become memory only through typed, scoped, provenance-aware paths, not through raw assistant-response mirroring.

User-authored memory should outrank assistant-generated prose and app-authored structured memory when they conflict. Work capsules, task continuity updates, generated document metadata, skill notes, and typed tool outcomes may be useful structured memory inputs, but they are lower authority than explicit user-authored durable statements, corrections, deletions, and review resolutions. When the user changes or corrects a remembered fact, active personalization should reflect that immediately while maintenance reconciles older structured records later.

Admitted material should pass through Memory Intake Normalization before durable memory writes. Normalization should produce a clean remembered statement with category, scope, reason, and provenance rather than write a raw user sentence or assistant response. The normalizer may be model-assisted, but the output should be constrained and locally validated. It should preserve user meaning, choose one primary category, and defer to maintenance when the statement, category, or scope is ambiguous.

Memory Scope should be first-class and separate from Memory Profile Category. Category is for human grouping and maintenance semantics: About You, Preferences, Goals & Ongoing Work, or Constraints & Boundaries. Scope is for applicability: where AlfyAI may use the remembered fact. The fixed scope set is global, project, conversation, and document. Scope assignment should use the narrowest confident scope; global scope is reserved for clearly user-wide remembered facts. Project scope attaches to the Project Folder when one is present, otherwise to confirmed Project Continuity. This prevents project-, conversation-, or document-specific memories from leaking into global personalization while still allowing related work surfaces to share the right context. Free-form client, team, topic, or arbitrary entity scopes should wait until Memory Rework Telemetry proves the fixed set is insufficient.

Document content needs a stricter boundary because uploaded files are often evidence, references, receipts, tax papers, generated drafts, examples, or documents the user is merely working with. Document-Sourced Context is not user-truth merely because it exists in the Knowledge Base or was attached to chat. Uploaded document contents should not sync directly into Honcho persona or session memory by default. A fact found inside a document should stay document evidence unless the user explicitly frames it as something AlfyAI should remember about them, their preferences, their constraints, their goals, or a specific document workflow. Document-Scoped Memory is for facts about a document or document family, such as how to interpret, edit, revise, or reuse it; it is not a bucket for every fact extracted from that document.

Document Memory Admission should be strict for immediate writes and observant for long-term learning. Immediate admission requires explicit user-authored durable intent about how AlfyAI should remember, treat, edit, interpret, revise, or reuse a document or document family. Durable language includes phrases such as "remember," "always," "from now on," or "for this document family." Repeated document workflow behavior may produce Memory Rework Telemetry, dirty state, and later maintenance or Guided Memory Review, but it should not silently become admitted durable memory. This allows AlfyAI to notice recurring document patterns without deciding on its own that document contents, one-off edits, or repeated edits are now user-truth.

Explicit memory intent should not require memory-specific controls in the ordinary chat interface. The Memory Rework Update should not add save-to-memory buttons, memory toggles, message-level memory editors, or review controls to the main chat composer or message stream. Users may express durable memory intent through normal chat language, and they may maintain memory through the dedicated Memory Profile and Guided Memory Review surfaces, but ordinary chat should continue to feel automatic rather than manual.

Category selection should follow stable precedence rather than free-form classifier taste. Constraints & Boundaries should win for hard rules, "never" or "must" language, sensitivities, deadlines, security, legal, or cost limits. Preferences should cover softer defaults and response or workflow tastes. Goals & Ongoing Work should cover explicit user-shared plans, objectives, and ongoing work the user would expect remembered. About You should cover stable personal or contextual facts, owned things, roles, and background. A remembered fact should not be duplicated across categories just because it has several aspects.

Memory review should be small, plain-language, and decision-oriented. A review item should ask a question such as "Which should AlfyAI remember?" and offer clear actions like confirm, correct, suppress, or keep inactive. It should not expose raw Honcho rows, technical contradiction diffs, maintenance logs, source inventories, or a durable-memory table as app UI.

The Memory Profile should be approachable first. Opening the Knowledge Base from navigation should default to Memory Profile rather than restoring Documents as the last selected tab. The default view should be an Active Memory Profile View: users should see and edit curated Memory Profile Items, the shorter active profile AlfyAI may actually use for personalization. Normal categories should show only currently active usable items, excluding deferred, blocked, expired, suppressed, deleted, and review-needed memory. It should not need explanatory feature copy such as "used by chat"; the profile structure and actions should make its purpose clear. It should refresh automatically when the user opens the Knowledge Base, so the primary profile surface does not need a generic reload icon or auto-refresh status sentence. If the active item count is shown, it should be a compact numeric badge rather than quota or status text. Repeated Memory Profile and Guided Memory Review actions should use familiar icon buttons with accessible names and tooltips on card, list, modal, and desktop detail surfaces; examples are check for use or save, pencil for edit, X for cancel or do not remember, and trash for delete. The item detail delete action should live in the same action row as save and cancel rather than a separate text-heavy danger control. Default cards should hide global-scope labels because they add visual noise; narrower scope labels such as project, conversation, or document should remain visible because they change where the memory applies. Item details should still show the full scope. Partially filled categories should not show filler empty states such as "no more active items"; a quiet empty state belongs only when a whole category has zero active items. Users should not have to manage hundreds or thousands of extracted chat-round memories. It should not provide a general inactive/history tab or browsable list of deleted, suppressed, expired, superseded, or inactive memories. Honcho supplies peer cards, conclusions, context, and representations; app-owned profile metadata supplies user-facing state such as active, corrected, suppressed, expired, or needs review. The profile should use a small, meaningfully distinct category set: About You, Preferences, Goals & Ongoing Work, and Constraints & Boundaries. It should not expose Focus Continuity, task memory, inferred work continuity, or backend continuity buckets as Knowledge Base memory sections; backend continuity can still help chat, folders, projects, and routing. Recently updated items should appear inside normal categories rather than in a separate recently-updated section. Confidence, freshness, provenance, and cleanup status should stay behind details unless the user needs to act on them. Provenance and edit surfaces should still be curated, human-readable UI rather than raw memory rows, Honcho dumps, markdown blobs, technical tables, salience/debug columns, or Honcho conclusion IDs. Item details should show one short why-this-is-remembered summary and 1-3 human-readable source chips by default, with a secondary source expansion for deeper inspection. The extended review modal should stay a compact row list with one question, one reason sentence, and icon actions per item; source evidence stays behind details. Provenance should be available behind details for inspection, but the default view should explain what AlfyAI knows, not where every extracted fragment came from.

Account-level reset should reuse the existing Clear Memory and Knowledge privacy control. The Memory Rework Update should preserve that action's ability to remove remembered context, Knowledge Base content, document-derived context, continuity state, and stored evidence traces while keeping chat conversations and the account. It should also clear any new Memory Profile projection metadata, review items, conflict blocks, intake dirty state, maintenance state, user-linked telemetry, and Honcho identity state that could otherwise rehydrate memory after reset. Non-identifying aggregate telemetry counters may remain only when they cannot identify the user or reconstruct remembered content. It should not introduce a second memory reset button in the Memory Profile or ordinary chat UI.

The Knowledge Base tab split should preserve ownership boundaries. Normal navigation to Knowledge Base should open Memory Profile first. Documents should be the second tab and keep upload, search, preview, workspace, download, and document-management behavior. Explicit document workflows such as global-search handoff, document open, upload follow-up, or a document deep link may activate Documents directly, but this should not make Documents the remembered default for ordinary Knowledge Base navigation. Documents should not expose memory review, memory edit, or save-to-memory controls; document-derived memory still enters through the Memory Intake Gate and maintenance rules.

Clear Memory and Knowledge should advance a durable Memory Reset Generation. Memory Profile projection rows, dirty ledger entries, review items, conflict blocks, maintenance slices, and retry work should be scoped to the current generation. Any in-flight or later retrying memory work that began under an older generation must discard its output and must not recreate profile, review, dirty, telemetry, or Honcho-derived memory state after reset. Quiescing active maintenance remains useful, but generation matching is the durable guard against old async work rehydrating cleared memory.

Memory Source Authority should follow the same UI rule as provenance. Item details may explain whether a remembered item came from a user-authored statement, review resolution, structured work output, maintenance reconciliation, or Honcho-derived memory. Ordinary Memory Profile cards should not show authority badges, source hierarchy labels, or trust-score UI by default. The default profile should stay focused on the remembered statement and actions; authority detail exists for inspection and debugging.

Guided Memory Review should be optional but discoverable. Persistent review items should be durable records created by memory maintenance, not regenerated from current conflicts on every Memory Profile refresh and not created by ad hoc chat-turn logic. Repeated conflicts or uncertain evidence should dedupe by user-facing review subject rather than raw Honcho rows, exact text, or source chats; new evidence for the same open subject should attach to the existing item. Review items should use a simple lifecycle: open, resolved, or obsolete. Open items need user attention; resolved items record the user decision needed to update the Memory Profile Projection and guide reconciliation; obsolete items are no longer relevant because newer evidence, expiry, merge, deletion, suppression, or reconciliation removed the need to ask. Review resolution should use three shared meanings: use this remembered fact, edit the remembered fact, or do not remember this subject. UI labels may vary by review item type, but guided review should not ask users to choose a keep-inactive, archive, source-delete, or technical reconciliation state. The Memory Profile should show a dedicated, full-width Needs Review area above the normal profile categories when unresolved review items exist, and the Memory Profile entry point should carry a small visible signal. The Needs Review area should be non-scrollable and show at most three review items; additional items should open in an extended modal view. Review items should not be mixed inline into the normal categories. It should not become a separate inbox, a settings-only surface, or a global notification. Normal chat should not be interrupted just because review work exists. Chat may ask a direct clarification when the current answer depends on unresolved remembered information, but durable review-item creation remains a maintenance responsibility; otherwise the user can review memory on their own time.

User actions from Guided Memory Review take immediate precedence in the Memory Profile Projection and active personalization. Corrections, deletions, and suppressions must be effective on the next chat turn even if Honcho still returns older remembered facts. Durable Honcho cleanup is projection-first and evidence-conservative: when the affected remembered fact has a Safe Memory Match, maintenance may reconcile the correction, deletion, or suppression with the memory authority by deleting or replacing matched remembered evidence. When provenance is partial or ambiguous, maintenance should not mutate uncertain backing memory; the user-facing correction, deletion, or suppression remains authoritative for active use while deeper reconciliation waits for safer evidence.

The Memory Profile Projection should be durable and materialized by the app as a thin read model. It should keep stable item identity and active-use state across Knowledge Base refreshes and chat turns. Memory Profile Item identity should be app-owned and derived from the normalized remembered statement, category, scope, and provenance relationship rather than from a single Honcho conclusion ID. One Memory Profile Item may aggregate multiple provenance links when several Honcho conclusions, user statements, review decisions, source chats, document rules, or structured work outputs clearly support the same remembered fact. User-authored merge precedence decides merge winners: user-authored edits, review decisions, suppressions, and deletions survive lower-authority Honcho-derived or structured memory, and conflicting user-authored states go to Guided Memory Review instead of being auto-merged. Splits should be conservative: if one item clearly contains multiple memory slots, maintenance may create child items with only the relevant provenance links, while preserving parent user-authored state and never reviving deleted or suppressed evidence into active children without new explicit user-authored evidence. Honcho IDs may be kept as provenance pointers for reconciliation, but they should not be the user-facing item identity because edits, merges, splits, and Honcho rewording must not break next-turn-effective profile state. Rebuilding the profile only from live Honcho output is not sufficient because edits, deletions, suppressions, expiries, conflict blocks, and review decisions must remain effective on the next turn even when Honcho has not reconciled yet. Live Honcho refresh may update or reconcile the projection, but it is not the projection itself and cannot override active-use state directly. This projection state should stay thin: curated statement, category, scope, active-use status, provenance pointers, safe Honcho identifiers when available, and timestamps. It should not become a second semantic memory engine, transcript mirror, or local replacement for Honcho.

Legacy Memory Migration should be a durable, lazy, per-user background migration rather than a full unbounded post-deployment sweep. It may use LLM-assisted pruning, but only through constrained structured decisions with local validation. The migration should be aggressive about active-use quality and cautious about information loss: only high-confidence, non-junk, category- and scope-fitting legacy memory becomes active in the Memory Profile Projection. Legacy material with unknown or ambiguous Memory Scope should not become active by default; it should become Preserved Legacy Memory or Guided Memory Review depending on user impact. The amount of active legacy material should be governed by an Adaptive Active Memory Budget, not a fixed product item count. The budget should respond to context pressure, relevance, source authority, scope, confidence, category balance, contradiction risk, and telemetry. When memory pressure is high, AlfyAI should merge, compact, expire active use, or move lower-priority material into Preserved Legacy Memory. User-authored corrections, hard constraints, and high-confidence scope-relevant memories should outrank softer, lower-confidence, or globally over-applied preferences. Internal operational caps may exist for safety, but they should not become the product definition of how much AlfyAI can remember. Memory pressure should remain internal maintenance and telemetry state; the default Memory Profile should not show profile-pressure warnings, memory quotas, or technical capacity indicators.

Potentially valuable but uncertain legacy material becomes Preserved Legacy Memory for future reconciliation or capped Guided Memory Review instead of being deleted merely because it is not active. Preserved Legacy Memory is a state in the same memory pipeline, not a second profile or user-browsable inactive memory store. It should not appear in the default Memory Profile, enter Active Memory Profile Context, or shape ordinary personalization. It may be used by bounded maintenance for merge, supersession, expiry, review, or deletion decisions, and by explicit history or source questions without becoming a hidden personalization bypass. Reconciliation of the preserved backlog should be trigger-based rather than comprehensive: related profile edits or deletions, new same-topic memory, detected conflicts, stale Memory Profile refreshes, or spare scheduled maintenance capacity may trigger bounded revisits, but the system should not try to process every preserved legacy item merely because it exists. Permanent deletion still requires the Automatic Junk Deletion Gate. The user-facing UI should stay minimal, such as a simple "preparing memory profile" state in the Memory Profile; it should not become a migration wizard or raw legacy review table. While migration is pending, ordinary chat should use the last valid projection or a reduced profile rather than continuing to treat raw noisy legacy Honcho output as current active personalization.

Chat prompt context should use Projection-Gated Memory Access. Active personalization should come from pre-filtered Active Memory Profile Context derived from the same active Memory Profile Projection the user can inspect. The Baseline Memory Profile is only the compact default slice of that context, not a second profile. It should use the same active usable item set shown in the normal categories of the Active Memory Profile View, while omitting UI-only details. Deleted, suppressed, expired, conflict-blocked, review-needed, preserved legacy, and ambiguous-scope material should be filtered out before ordinary prompt assembly rather than sent to the model with status labels and instructions to ignore it. Honcho remains the memory authority and substrate, but raw Honcho context should not directly enter ordinary personalization or override user-visible edits, deletions, suppressions, expiries, deferrals, conflict blocks, or review blocks. This keeps the Memory Profile intelligible: what the user sees as actively remembered is the profile AlfyAI may use for ordinary personalization.

The Memory Context Tool may still retrieve Honcho-backed persona context, historical chats, and documents when they are relevant to the current turn, even when those facts are not active Memory Profile Items. That retrieval path must not become a hidden personalization bypass. Deleted or suppressed profile memory should not be returned as remembered context through memory recall, even for history/source questions, although exact source-record retrieval may still find original chats or documents if those records still exist. Expired, inactive, preserved legacy, or ambiguous-scope material may appear only as Historical Memory Evidence for explicit history, source, document, or evidence questions. Historical Memory Evidence should be compact: a short excerpt or summary, source kind, source title or date when available, evidence framing such as historical-only, source-record, or unresolved-conflict, and enough provenance to cite or inspect. It should not include full inactive profile tables, confidence/debug scores, raw Honcho dumps, deletion/suppression internals, or instructions asking the model to decide whether it is current truth. Conflict-blocked or review-needed material may appear only when the user asks about the conflict or source history itself, and it should be framed as unresolved rather than current profile truth. Active Memory Profile Context controls what AlfyAI treats as current profile truth.

The default edit action should be a full direct edit of the remembered statement, not a guided field editor or category-specific wizard. If AlfyAI cannot make the edited statement effective for the next chat turn, the item should not expose an edit action. When the edit can be applied immediately in the Memory Profile Projection, Memory Edit Classification should decide whether the change is a same-item correction, a replacement item, or an ambiguous rewrite that needs plain user confirmation. That classification should compare the normalized memory slot, including category, scope, subject, predicate, and authority relationship, rather than text length or character similarity. Durable Honcho reconciliation may still happen afterward by deleting and replacing remembered evidence only when a Safe Memory Match exists.

The default item-level destructive action should be user-facing Memory Profile Deletion, not a technical choice between delete and suppress. When a user deletes a Memory Profile item, AlfyAI should immediately remove it from active use and keep it out of the next chat turn. If the remembered evidence can be safely matched to durable records, those records should be permanently removed. If matching is partial or ambiguous, the item still stays out of active use while maintenance continues reconciliation. This action does not delete the original chat messages, documents, account, or workspace data where the information may have first appeared.

This immediate path should stay intentionally small. It exists to honor explicit user edits and suppressions in active use; it should not become app-owned persona synthesis, semantic clustering, dreaming, or a replacement for Honcho. Honcho should continue doing the hard memory work. AlfyAI's app-owned layer should serve the curated profile, protect user corrections, and reconcile safely over time.

Memory should remain one pipeline rather than split into separate "hot" and "cold" memory systems. Immediate work may apply already-authoritative state changes, such as project or folder continuity, working-document continuity, accepted intake decisions, and user-authored profile corrections or deletions. Expensive maintenance may then reconcile, deduplicate, expire, clean, trigger Honcho Dreaming, or create Guided Memory Review items later. The delayed maintenance queue is not a source of prompt truth; prompt context should read from the canonical continuity, profile, Honcho, and Context Selection boundaries.

Expiration means active-use expiry, not automatic deletion. It should be conservative and evidence-based rather than simple age-based decay. Expired remembered facts should stop shaping the Memory Profile, Baseline Memory Profile, and active personalization unless refreshed or confirmed. Maintenance should expire active use when newer user statements clearly supersede older facts, time-bound facts have passed, contradictions require review, stale facts would cause bad personalization, or the user edits or deletes related profile items. Stable facts such as names, long-term preferences, durable writing style, important boundaries, and durable personal context should not expire merely because they are old. Historical evidence may remain until the user explicitly asks AlfyAI to forget or suppress it, or maintenance can safely remove junk or obsolete duplicates.

Same-scope contradictory memories should not be handed to the model as ordinary active prompt context. When maintenance can safely determine supersession, the older fact may leave active use through Memory Active-Use Expiry. When AlfyAI cannot safely decide which fact is current, a Memory Conflict Block should keep the conflicting items out of ordinary personalization and create or update Guided Memory Review when user authority is needed. This block is item-level and scope-aware; it should not suppress a whole category or treat different-scope memories as contradictions merely because their wording overlaps. Conflict review should appear in the dedicated Memory Needs Review area rather than as blocked items inside normal Memory Profile categories.

Automatic permanent deletion should be reserved for Junk Memory that passes an Automatic Junk Deletion Gate. The gate is a policy used by intake and maintenance, not a separate cleanup system. It should require safely identifiable junk, exact remembered evidence to target when deletion is involved, no user-meaningful veto, and no support for an active Memory Profile item. Examples include malformed extraction residue, accidental technical artifacts, boilerplate interaction summaries, and meaning-preserving duplicates. Meaningful but problematic memories should not be silently deleted just because they are old, contradictory, sensitive, stale, or inconvenient; those cases belong to active-use expiry, Memory Profile Deletion, or Guided Memory Review. The gate should run at memory intake and during background maintenance so junk does not accumulate again. Intake should block only obvious junk before it reaches memory authority. Background maintenance may clean more broadly because it can inspect duplicates, supersession, provenance, and active-profile support.

Honcho Dreaming should be treated as an internal memory-reconciliation tool, not a user-facing Memory Profile state. AlfyAI may use it after profile edits, deletions, cleanup, supersession, and larger memory batches to help the underlying memory substrate catch up. The Memory Profile Projection should provide the immediate next-turn-effective truth; users should not have to understand whether background dreaming or reconciliation has finished before trusting the profile.

Memory authority failures should be projection-first. When Honcho refresh, delete, or reconciliation fails, the durable Memory Profile Projection remains the user-facing active truth. User edits, deletions, suppressions, and review decisions should still succeed locally when their visible projection change is saved; failed Honcho cleanup becomes retryable dirty work and Memory Rework Telemetry. Background Honcho cleanup failures should not appear as user-facing action errors unless the visible projection save itself failed. If the durable projection cannot be read, the Memory Profile should show an ordinary load failure or retry state rather than falling back to raw Honcho output.

Routine maintenance should be bounded by a durable typed Memory Dirty State Ledger, cooldowns, and work budgets so chat activity does not burn the server. The ledger should coalesce repeated account-level signals such as stale projection, deferred intake, profile-action reconciliation, possible conflict, possible duplicate, legacy migration, Honcho reconciliation, and review generation. It may store reason, scope, stable identifiers, counts, and timestamps, but not raw candidate memory text. Chat turns, Memory Profile Refresh, profile actions, legacy migration, and Memory Authority Fallback should produce typed dirty work instead of directly running expensive reconciliation. A single Memory Maintenance Scheduler should claim expensive dirty work by user, priority, cooldown, and budget; multiple active chats for the same user should coalesce into shared dirty work instead of independent reconciliation runs. Expensive reconciliation should run in Bounded Memory Reconciliation Slices rather than trying to clear all dirty work for a user in one pass. A slice should prioritize user-facing correctness, prompt safety, profile quality, legacy migration or preserved backlog, then opportunistic junk cleanup; if the budget is exhausted, remaining dirty ledger entries stay pending for later work. Cooldowns and budgets gate expensive cleanup, expiry, deduplication, review generation, and durable reconciliation; they should not delay active project or folder continuity reads and writes. Opening the Knowledge Base should run a cheap Memory Profile Refresh from the current durable projection, not synchronous Expensive Memory Reconciliation. If the projection is stale or dirty, the read may enqueue bounded background maintenance, but it should not wait for LLM pruning, deduplication, Honcho reconciliation, or Honcho Dreaming before rendering the Memory Profile. User actions such as edit, delete, suppression, and review resolution should take immediate effect in active use, advance the affected Memory Projection Revision, then enqueue targeted reconciliation rather than broad maintenance by default. Maintenance output should apply only when the Memory Projection Revision still matches the state it read; stale output should be discarded or retried against the current projection so background work cannot overwrite newer user-facing memory state. Half-finished maintenance should leave uncommitted or unfinished dirty work pending. Chat turns may mark memory dirty and perform Immediate Memory Admission for high-confidence material, but they should not trigger full Expensive Memory Reconciliation by default.

For the current small deployment, defaults can be conservative but useful: the memory/Honcho LLM path uses Qwen 3.6 35B on vLLM with an RTX PRO 5000 Blackwell 48GB GPU, measured around 130 tokens per second at concurrency 1, while chat usually runs on third-party models or a different GPU. With four total users, normally one active user, roughly 350 messages and 10 million tokens in May, around 150 conversations per month, sparse document uploads, and one extreme legacy user around 1600 raw memories, the baseline should be `DERIVER_WORKERS=2`, one global expensive app-level memory LLM job at a time, one expensive maintenance slice per user at a time, scheduler wake or poll about every five minutes when dirty work exists, a normal slice soft cap around 60 seconds and hard cap around 120 seconds, and a normal per-slice LLM input budget around 25k tokens. Legacy migration should use the same bounded scheduler lane as other expensive work rather than a special accelerated catch-up path; waiting longer is acceptable because migration is asynchronous and should not change visible correctness once the durable projection is available.

Bounded maintenance slices should also have count-based batch limits so token and time budgets are not the only safety valves. Bootstrap defaults should inspect at most 50 candidate memories per slice, apply at most 20 projection mutations per slice, attempt at most 10 Honcho delete or replace calls per slice, create at most three new review items per slice, and trigger at most one Honcho Dreaming run per user per day, only after meaningful batched changes. A slice should stop when it reaches any configured time, token, candidate, projection-mutation, authority-call, review-item, or dreaming limit.

Review burden should be capped more tightly than background maintenance. The default should create at most three new review items per bounded maintenance slice and keep at most twelve open review items per user. When the cap is reached, additional ambiguous material should become inactive or Preserved Legacy Memory rather than more review work. Higher-impact review work may obsolete or replace lower-priority open review items, but it should not exceed the open review cap. The visible Memory Profile still shows at most three review items in the Needs Review area, with overflow in the extended modal.

Automatic memory decisions should use coarse confidence bands rather than pretending classifier scores are precise truth. Bootstrap defaults should require roughly `0.90+` confidence plus clear category and scope for auto-active admission; roughly `0.85+` confidence plus junk-pattern support and no active-profile support for automatic junk rejection; roughly `0.85+` confidence plus same-scope newer evidence for automatic active-use expiry or supersession; roughly `0.55-0.89` confidence for review when user impact is meaningful and review caps allow it; and preservation or inactive use for meaningful material below active confidence, with ambiguous scope, or overflowing review caps. Permanent deletion should still require the Automatic Junk Deletion Gate rather than a confidence score alone. These bands must not override source authority: user-authored corrections, deletions, suppressions, and review decisions win over lower-authority classifier output.

Active profile pressure should use invisible internal ceilings, not user-facing quotas. Bootstrap defaults should target roughly 80 active Memory Profile items, use an internal hard ceiling around 150 active items, and treat 20-30 active items per category as a soft balance target rather than equal quotas. Constraints & Boundaries may exceed its soft category target when the items are durable hard constraints. When pressure is high, maintenance should merge or compact first, expire active use second, move lower-authority or lower-relevance items to Preserved Legacy Memory third, and create review items only when user judgment is needed and review caps allow it. The Memory Profile should not show memory-full messages, item quotas, or pressure warnings.

Memory Rework Telemetry is part of the product design. AlfyAI should record enough operational evidence to evaluate whether the system can run unattended: which intake candidates were accepted or rejected, what maintenance actions were taken, how many review items were created and resolved, how often users corrected or deleted profile items, and when active memory was included in or blocked from prompt context. Telemetry should use fixed event families instead of free-form labels: intake, active profile projection, prompt use, maintenance, guided review, profile action, reset or forget, and error or fallback. Telemetry should be privacy-preserving by default: decisions, categories, reasons, counts, statuses, and stable identifiers are appropriate default telemetry, while raw remembered text and raw chat excerpts require a narrow explicit debug mode. User-linked telemetry should clear with Clear Memory and Knowledge. Non-identifying aggregate counters may remain only when they cannot identify the user or reconstruct remembered content. Telemetry should be backend/log-only by default; the Memory Rework Update should not add a user-facing or admin-facing telemetry summary view. A future focused admin summary may be added after collected telemetry proves which metrics are useful. Telemetry should measure the behavior of the memory system without becoming a second sensitive memory store.

**Considered Options**

- Keep memory fully background and autonomous.
- Mirror every chat message into Honcho and rely on later cleanup.
- Mirror assistant-generated answer text into durable memory.

We chose a complete Memory Rework Update with Guided Memory Review, a Memory Intake Gate, Memory Intake Normalization, and first-class Memory Rework Telemetry because fully autonomous memory makes stale or contradictory facts hard to correct and forces the model to reason over competing memories at answer time. Mirroring every message into Honcho creates a noisy substrate that maintenance can only fight after the fact. Fully manual memory management would turn memory into user work and recreate the unusable maintenance-table problem. Separate hot and cold memory systems would create drift, and a long-lived pending-candidate store would become another memory substrate to maintain. Free-form classifier inference would recreate noisy memory with nicer labels. A partial v1 would leave the system dependent on manual retuning before enough telemetry exists. Adding observability later would hide whether the unattended loop is working during the period when the design most needs evidence. A single memory pipeline with immediate authoritative writes, constrained normalization, delayed reconciliation, and telemetry from the start keeps memory mostly invisible while giving the user control where the system lacks enough authority.
