# Unify file production behind one service boundary

AlfyAI file production will be exposed to the model as one user-facing capability and owned server-side by a dedicated file-production boundary. This is a production v1 capability, not a prototype path: reliability, observable failures, recovery, durable job lifecycle, and coherent user-facing behavior are part of the initial design. The assistant supplies semantic content for document-like outputs, AlfyAI chooses the renderer or runtime, generated outputs enter prompt context only through Context Selection, and storage/listing/download plumbing remains separate from production orchestration. Production jobs will be durable records separate from generated-file storage, because a job can queue, fail, retry, be cancelled, or produce multiple files while `chat_generated_files` should represent files that actually exist. File production execution is durable background work: chat streaming may create or announce a job, but the SSE connection does not own rendering completion. V1 uses an in-process durable worker backed by SQLite job claims and heartbeats, not Redis, BullMQ, a separate worker service, or another queue system; the database state remains the source of truth so a separate worker process can be introduced later without changing the product contract. The default worker runs one active file-production job per process and claims eligible queued jobs FIFO by creation time, keeping resource use predictable while preserving a configurable concurrency cap for later. Each worker process owns a generated `workerId`; running attempts store `workerId`, `claimedAt`, and `heartbeatAt`, and heartbeat/finalization must match the current attempt and claiming worker so late stale-attempt results cannot overwrite newer job state. The worker scheduler starts once per process from server bootstrap/maintenance, wakes lazily when jobs are created or retried, drains eligible work, and idles instead of running a tight empty-queue loop; startup runs stale-attempt recovery and wakes once to drain already-queued jobs, while shutdown stops new claims and relies on heartbeat recovery for interrupted attempts. Generated-document source is untrusted structured data even when produced by the model: renderers escape user/model text by default, raw HTML is rejected in v1, image/URL inputs go through server-controlled validation, generated HTML output is produced by AlfyAI rather than accepted as model-authored markup, and generated HTML previews are served with restrictive CSP/sandboxing. Produced-file downloads and previews validate stored MIME type, filename extension, declared output type, and bytes where practical; mismatched or unsupported content is not previewed, and downloads use sanitized `Content-Disposition`. V1 image fetching allows only `https` URLs, internal artifact/generated-file references, and size-limited `data:image/...;base64` URIs; it rejects `http`, `file`, localhost/private IPs, link-local IPs, non-image MIME types, oversized images, and redirects to disallowed final targets. Program/sandbox mode remains network-disabled by default and receives any needed files/images only through validated internal references, so generated code does not become a second web/image fetching subsystem. V1 limits can be generous for a small trusted user set, but expensive dimensions still need explicit server-side bounds, stable error codes, localized messages, and measured diagnostics so oversized requests fail predictably; effective limits flow through the existing runtime config/admin settings path and are persisted in diagnostics when they fail. Renderer and sandbox timeouts stay distinct retryable error classes even when their default duration matches, because source/template/render failures and user/programmatic generation failures need separate diagnostics and tuning. When a static size/count/page/table/chart/image/output limit fails, partial outputs must be discarded before storage/linking and the job must fail with a non-retryable limit error unless the source/request changes. In v1, visible queued/running job cards update through lightweight polling of persisted job state rather than a dedicated file-production SSE channel. A single job may produce multiple files from one source, and the UI should keep them grouped under one production card/job instead of treating each file as an unrelated chat event. Job cards must be recoverable after refresh, reconnect, and failure; generating, failed, and cancelled states should come from persisted job state rather than temporary stream-only placeholders. Failed jobs should retry under the same job identity with attempt history and idempotency keys, so one user request does not create duplicate cards or duplicate files during stream/tool retries. Failed jobs are not automatically retried in v1; retryable failures surface an explicit user retry action that creates a new attempt under the same job. Queued or running jobs should support best-effort user cancellation that persists `cancelled` even when renderer/runtime cleanup completes asynchronously afterward. Stale running attempts should be recovered by heartbeat timeout and converted into retryable infrastructure failures so cards cannot stay stuck forever. File-production input validation and source normalization must run before renderer/runtime work; malformed source becomes a durable failed job with localized validation detail instead of an invisible or ambiguous runtime failure. User-visible card states and errors should be short, actionable, and localized through the existing English/Hungarian i18n system; technical diagnostics belong in compact details or server logs, not raw chat text. File-production observability should use one compact orchestration vocabulary, with `[FILE_PRODUCTION]` for job lifecycle and existing lower-level prefixes such as `[SANDBOX]`, `[CHAT_FILES]`, `[MINERU]`, and `[HONCHO]` for their own boundaries. Not every generated file is a generated document: raw data exports, code, images, and bundles should stay downloadable files unless user intent makes them working-document material. The model may provide document intent as a hint, but server-side classification is authoritative so bad tool arguments cannot corrupt document family, versioning, workspace, or context-selection behavior. Document-like jobs will be source-first: persist canonical generated-document source as versioned JSON in the `generated_output` artifact metadata, store a deterministic readable projection in `generated_output.contentText` for retrieval/context/Honcho/search paths, and link rendered binaries back to that artifact. The same source can render v1 `pdf`, `docx`, and `html` outputs, with PDF as the primary reliability target; spreadsheet and CSV outputs stay in generated-file program/data production. The source schema will be constrained to semantic blocks and template-safe options, not arbitrary HTML, CSS, absolute positioning, or PDF drawing commands. Tables are a first-class block with production pagination and formatting behavior, while merged cells and nested tables stay out of v1 unless separately designed. Do not create a second parallel source artifact for the same generated document. Sandboxed program execution remains available for genuinely programmatic/raw outputs, but it is not the default document/PDF production path. MinerU extraction remains a fallback/readability path for binaries, while refinement, rerendering, and context selection should prefer the semantic source when it exists. The v1 UI will keep generated-file cards as the primary chat surface, making them richer, persistent, and actionable while deferring broader workspace and downloader redesign.

Implementation should move in small production slices. Each slice must be independently testable and verifiable, leave the system usable, include appropriate fallback/observability, and remove the specific old behavior it replaces rather than accumulating parallel paths.

The existing `generate_file` and `export_document` Langflow tools and endpoints should be treated as obsolete migration targets, not compatibility surfaces. Once the unified file-production tool/node and endpoint are ready, remove and disable the obsolete tools/routes instead of keeping shims. Model guidance, AGENTS files, README/setup notes, and other agent-facing documents must be updated in the same migration so no instruction path continues to point at the old generation system.

**Considered Options**

- Keep separate model-facing PDF export and sandbox file-generation tools.
- Make sandboxed code generation the universal production path.
- Unify file production behind a dedicated server service while keeping renderers, sandbox execution, storage, and context selection as separate boundaries.

We chose the unified boundary because the current split makes simple file requests brittle, especially PDFs, and forces the model to choose between implementation paths. Server-owned production preserves style control, gives generated outputs a clean context-selection contract, and lets document source, rendering, persistence, and preview evolve without reintroducing route-local or tool-local file workflows.
