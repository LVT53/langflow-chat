How to read this report

A candidate is worthwhile when the current module is shallow at the caller interface, leaks implementation knowledge across seams, duplicates a seam, or creates hidden coupling. A candidate is not worthwhile just because a file is long.

1. Split Memory Profile Internals Behind the Existing Seam

Highest leverage Hidden coupling

Files involved

  • src/lib/server/services/memory-profile/index.ts (3329 lines)
  • src/lib/server/services/memory-profile/intake.ts
  • src/lib/server/services/memory-maintenance.ts
  • src/lib/server/services/chat-turn/context-selection.ts
  • src/lib/server/services/normal-chat-context.ts
  • src/lib/server/services/normal-chat-control-model.ts

What leaks

One module exports profile projection reads, item mutation, reset-generation checks, Guided Memory Review, dirty-ledger reconciliation, legacy curation, prompt formatting, and telemetry. Fallow shows this module in most circular-dependency paths involving memory, chat context, control-model work, and maintenance.

Problem

The public interface gives callers too many reasons to import the catch-all module, so implementation knowledge travels through unrelated chat, memory, and maintenance paths.

Solution

Keep the current facade for compatibility, but move implementation clusters into internal modules. Let hot callers import narrower modules only when that is the real seam.

Wins

  • Break several Fallow cycles without behavior changes.
  • Increase locality for review and dirty-ledger fixes.
  • Make memory tests target the real interface for each slice.

ADR callout: ADR 0033 already names Memory Profile Projection, Guided Memory Review, dirty ledger work, legacy migration, and telemetry as distinct concepts. This refactor follows that design instead of re-litigating it.

2. Move Analytics Dashboard Read Model Out of the Route Adapter

Narrow win Leaky route

Files involved

  • src/routes/api/analytics/+server.ts (563 lines)
  • src/lib/server/services/analytics.ts
  • src/routes/api/analytics/analytics.test.ts

What leaks

The route imports db, schema tables, provider lookup logic, grouping code, timeline code, and mock dashboard payloads. It is an implementation module wearing a route adapter shape.

Before

  Route["+server.ts"]
  DB["db + schema"]

  Route --> DB
  Route --> Providers

After

flowchart LR
  Route["+server.ts adapter"]
  ReadModel["analytics/read-model.ts"]
  Writer["analytics.ts event ingestion"]
  DB["db + schema"]
  Route --> ReadModel
  ReadModel --> DB
  Writer --> DB

Problem

The route interface is small, but its implementation has durable analytics read logic that cannot be reused or tested without constructing a SvelteKit event.

Solution

Add getAnalyticsDashboardReadModel under the analytics module. Keep +server.ts to auth, parameter parsing, and JSON response.

Wins

  • Aligns with the repo rule that routes are adapters.
  • Gives admin and personal dashboard math a direct test surface.
  • Creates locality for future privacy and retention changes.

ADR callout: ADR 0029 says person-linked analytics must be removed during Account Erasure while anonymous aggregates may remain. A dedicated read model makes that privacy shape easier to verify.

Solution

Wins

  • Lower test setup for effective skill resolution.
  • Clearer seam between app-owned skill state and prompt-time use.

ADR callout: ADR 0009 says skills are app-owned product state with session behavior, settings, permissioning, and prompt integration. The current module implements that decision, but the implementation is too compressed for the number of reasons it changes.

UI locality Duplicated seam risk

Problem

The visible list has a broad interface and a broad implementation, so visual changes risk touching search, upload, and document-content behavior.

Solution

Move pure list derivation and file-kind mapping to tested TypeScript modules. Add a browser fetch module call for prompt artifact content. Keep the Svelte file focused on rendering and user gestures.

Wins

  • Direct tests for search scoring and sort stability.
  • Removes raw fetch from the visual file.
  • Prevents another document viewer or document fetch seam from appearing later.

ADR callout: ADR 0021 says preview runtime should own preview internals, while Working Document Identity and file serving stay elsewhere. This candidate follows that split by keeping list behavior separate from preview and durable document identity.

  • Faster tests for ranking and brief construction.
  • Less risk when adding a provider adapter later.

ADR callout: The AGENTS map says web-grounding.ts owns model-safe source and evidence shaping. This candidate keeps that public seam and deepens only the retrieval implementation under it.

Problem

The scheduler is a correct owner for timing, but it imports too much implementation. That makes unrelated feature modules part of the same dependency knot.

Solution

Extract cleanup-only step adapters that depend downward on schema and narrow helpers, while feature modules keep their user-facing interfaces. Do this only after candidate 1 reduces memory-profile fanout.

Wins

  • Attacks the known Fallow cycles directly.
  • Improves locality for reset and cleanup correctness.
  • Reduces chance that a file-production change loads memory maintenance.

ADR callout: The AGENTS map says memory-maintenance.ts owns scheduling and should not be bypassed. This candidate keeps that ownership and narrows the step implementation dependencies.

Suggested order

  1. Start with Analytics Read Model. It is small, route-adapter aligned, and easy to verify.
  2. Then split Memory Profile internals. It has the highest dependency leverage.
  3. Use the Memory Profile split to decide whether Maintenance Step Adapters are still needed or smaller than expected.
  4. Do Skills after that; it is product-important but has less immediate cycle pressure.
  5. Do Knowledge Documents and Web Research when you are already touching those areas.
