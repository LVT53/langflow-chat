## Oracle P0 Fix Assessment

The targeted two-number tune (`maxSearchQueries: 6 → 5`, `maxAcceptedWebSources: 16 → 28`) achieved the primary goal: **rejection ratio dropped from 72.9% to 45.1%** — a 27.8 percentage point improvement, well below the 50% threshold.

### How it works:
1. **Fewer queries (6 → 5)**: Less source overlap/duplication across queries, reducing the denominator of unique-but-low-quality sources.
2. **More slots (16 → 28)**: 75% more source acceptance capacity shifts the `source_cap` boundary outward. Sources that were previously cap-rejected now get accepted, while the quality filters (unusable_snippet, duplicate_url, unsafe_adult_content) continue to reject truly low-quality material.
3. **Combined effect**: 5 queries produce ~51 unique web sources. The quality filter rejects 23, leaving 28 for acceptance. No `source_cap` rejections needed — all quality-filtered sources fit within the 28-slot budget.

### Side effect: Higher truncation
With 28 accepted sources (vs 16 before), the writer prompt grows from ~59K to ~71K chars, resulting in 18.0% truncation (vs 1.53% before). This is a trade-off: more evidence at the cost of more aggressive prompt compression. The truncation cascade (levels 1-5) handles this via fact-per-card reduction and synthesis/outline trimming.

### Remaining concern
The higher truncation rate may impact report quality for overview jobs with many accepted sources. This could be addressed in a future task (e.g., raising `ATLAS_MAX_WRITER_PROMPT_CHARS` for the overview profile, or making truncation less aggressive when many sources are accepted). This is a separate concern from the rejection ratio, which is now solidly under 50%.
