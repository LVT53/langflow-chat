Chat Streaming Stability And Preparing Response Speed

Scan date: 2026-06-28. Focus: stabilize AI SDK/OpenAI-compatible streaming across DeepSeek V4, Xiaomi, Kimi K2.X, GLM 5.X, and Qwen 3.X model families, while making the app-side path before the first meaningful streamed event as fast as practical.

Evidence Base

Context7 AI SDK docs confirm that streamText().fullStream carries structured start, text, reasoning, source, file, tool, finish, error, abort, and raw parts. Context7 SvelteKit docs confirm +server.ts handlers return standard Web Response objects, so a streamed response can be returned as soon as the route has enough validated state. Local ADRs confirm AI SDK UI streams are the browser-facing seam and Normal Chat Model Run owns provider attempt and fallback policy.

1. Add a Stream Intake Fast Path

Highest leverage Hidden coupling

Files involved

  • src/routes/api/chat/stream/+server.ts
  • src/lib/server/services/chat-turn/stream-orchestrator.ts
  • src/lib/server/services/chat-turn/stream-completion.ts
  • src/lib/server/services/memory-profile
  • src/lib/server/services/file-production

What leaks

The route currently waits on getCurrentMemoryResetGeneration before returning the streamed Response. The orchestrator then snapshots file-production jobs before provider context preparation. These are durable completion facts, but their current location makes them part of the response-start interface.

Before

flowchart TD
  Browser["browser fetch"]
  Route["stream route parse, capacity, preflight"]
  Reset["memory reset generation read"]
  Response["return streamed Response"]
  Start["stream start prelude"]
  Jobs["file-production job snapshot"]
  Context["context-preparing"]
  Model["provider stream"]
  Browser --> Route --> Reset --> Response --> Start --> Jobs --> Context --> Model

After

flowchart TD
  Browser["browser fetch"]
  Route["stream route parse, capacity, preflight"]
  Response["return streamed Response"]
  Start["prelude plus preparation activity"]
  LazyFacts["lazy completion facts"]
  Context["context-preparing"]
  Model["provider stream"]
  Completion["complete stream turn"]
  Browser --> Route --> Response --> Start --> Context --> Model --> Completion
  Start -.-> LazyFacts -.-> Completion

Problem

The module interface says "start a stream", but the implementation performs completion-oriented reads before the stream can visibly start. That reduces perceived speed and makes unrelated storage latency look like model latency.

Solution

Create an internal stream intake module with a small interface: validate the turn, return the streamed Response, and hand lazy completion facts to the orchestrator. Move memory reset generation and file-production start snapshots into lazy promises consumed at completion.

Wins

  • Faster response headers and prelude.
  • Better locality for "preparing response" timing.
  • Cleaner distinction between intake facts and completion facts.
  • More precise phase timing for app-side delay vs provider delay.
ADR callout: Preserves ADR-0019 and ADR-0025 because the browser stream seam remains AI SDK UI frames; only the order of app-owned preparation work changes.

2. Deepen the OpenAI-Compatible Provider Adapter

High leverage Duplicated seam

What leaks

Provider family knowledge is split across request-body transforms, provider options, stream normalization, Xiaomi MiMo reasoning replay, and the model-run event adapter. DeepSeek, Kimi, Qwen, and MiMo are partly known; GLM is absent as an explicit family.

After

flowchart TD
  Adapter["provider adapter interface"]
  Profiles["family profiles: DeepSeek, Xiaomi, Kimi, GLM, Qwen, generic"]
  Request["request shaping"]
  Stream["stream shaping"]
  Events["neutral event mapping"]
  Adapter --> Profiles
  Profiles --> Request
  Profiles --> Stream
  Stream --> Events

Problem

The module is shallow for new model families: adding GLM 5.X or a Qwen 3.X quirk means touching several files and remembering which quirks belong before the AI SDK call and which belong after provider bytes arrive.

Solution

Introduce a single provider adapter interface that returns request transforms, provider options, stream transforms, reasoning/tool-call normalization rules, usage expectations, and retryable error classification for a provider family.

Wins

  • One seam for DeepSeek V4, Xiaomi, Kimi K2.X, GLM 5.X, and Qwen 3.X quirks.
  • Higher depth in tests: one adapter fixture can exercise request and stream behavior together.
  • Less provider-specific leakage into the generic model-run implementation.
  • Safer future version bumps when providers change OpenAI-compatible details.
ADR callout: Preserves ADR-0027 because Provider Models still identify model capability and runtime defaults; this only deepens the provider execution adapter.

3. Add a Provider Stream Fixture Harness

High leverage Shallow tests

Files involved

  • src/lib/server/services/normal-chat-model/index.test.ts

  • src/lib/services/ai-sdk-ui-stream-contract.ts

What leaks

Current tests cover generic AI SDK parts and compatibility helpers, but the interface for a provider-family stream is implicit. There is no compact place to replay real-ish DeepSeek, Kimi, GLM, Qwen, and Xiaomi chunks through the same implementation used in production.

After

flowchart TD
  Harness["provider stream fixture harness"]
  Families["family fixtures"]
  Adapter["provider adapter"]
  ModelRun["model-run stream"]
  Contract["AI SDK UI stream contract"]
  Families --> Harness --> Adapter --> ModelRun --> Contract

Problem

Stability depends on provider-specific stream shapes, but the tests do not give enough locality for adding a new family or version. The practical interface is "a provider emits odd OpenAI-compatible bytes"; tests should cross that same seam.

Solution

Add a fixture harness that feeds provider-family SSE and JSON completion samples into the adapter, model-run, and AI SDK UI stream contract. Keep fixtures small, anonymized, and version-labeled.

Wins

  • Fast local proof for new providers before live smoke checks.
  • Tests become the interface for stream compatibility.
  • More confidence around reasoning, tool-call, finish, usage, and abrupt-close behavior.
  • Lower blast radius when updating the AI SDK or provider transforms.
ADR callout: Preserves ADR-0026 by testing the Vercel AI SDK Normal Chat path, not reintroducing a second model runtime.

4. Make Context Preparation a Staged Module

Medium leverage Large interface

Files involved

  • src/lib/server/services/normal-chat-context.ts
  • src/lib/server/services/chat-turn/context-selection.ts
  • src/lib/server/services/chat-turn/streaming-normal-chat-model-run.ts

What leaks

prepareOutboundChatContext builds constructed context, prompt text, automatic compression, forced web prefetch, token budgets, and trace data in one sequential implementation. Callers only learn that it is "preparing"; they cannot see the stages or optimize independent work.

Before

  Budget["prompt budget"]
  Prepare --> Construct --> Prompt --> Compress --> Prefetch --> Budget

After

flowchart TD
  Plan["context preparation plan"]
  Base["base prompt and static guidance"]
  Context["constructed context"]
  Compression["compression decision"]
  Prefetch["prefetch decision"]
  Final["final prompt"]
  Activity["typed stage activity"]
  Plan --> Base
  Plan --> Context
  Base --> Compression
  Context --> Compression
  Compression --> Prefetch --> Final
  Plan --> Activity

Problem

The module has useful implementation depth but a coarse interface. That makes "preparing response" opaque and makes performance work depend on reading a 1,496-line file.

Solution

Keep the public function but give it an internal staged implementation with typed stage results and activity emissions. Run independent prompt/static work alongside constructed-context work where possible.

Wins

  • Better locality for context latency.
  • Clearer "preparing response" activity in the UI.
  • Safer changes to compression and prefetch sequencing.
  • Smaller focused tests around each stage.
ADR callout: Preserves ADR-0002, ADR-0003, and ADR-0006 because context selection still owns prompt-context construction and budgeting.

5. Deepen Stream Activity And Timing

Medium leverage Leaky interface

Files involved

  • src/lib/services/streaming.ts

What leaks

Phase timing, response activity, stream metadata, and browser timing are related but live in different shapes. The UI sees generic activity such as context-preparing, while logs track route phases, prelude, first upstream event, first thinking, and first visible token.

Before

  BrowserTiming["browser timing snapshot"]
  Metadata["terminal metadata"]
  UI["chat UI"]
  Orchestrator --> Metadata
  Activity --> UI
  BrowserTiming --> UI

After

flowchart TD
  Timeline["stream timeline module"]
  Server["server phase marks"]
  Activity["user-visible activity"]
  Browser["browser timing"]
  Metadata["terminal metadata"]
  UI["chat UI"]
  Timeline --> Server
  Timeline --> Activity
  Timeline --> Browser
  Timeline --> Metadata
  Activity --> UI

Problem

Performance work needs one mental model, but the implementation exposes several. That makes it hard to prove whether "preparing response" is route intake, context, provider connection, or delayed visible token emission.

Solution

Add a stream timeline module with a small interface for server marks, user-visible activity, terminal metadata, and browser timing correlation. Keep frame encoding in the existing AI SDK UI stream contract.

Wins

  • One place to measure first byte, first activity, first upstream event, and first visible token.
  • Better diagnostics for provider-family stability.
  • Fewer ad hoc strings for activity ids and phase names.
  • Cleaner tests for progress UI during slow providers.
ADR callout: Preserves ADR-0019 because the browser client runtime remains above transport parsing; the timeline module only deepens activity and timing locality.

Recommended First Grill

Start with Candidate 1, then Candidate 2. Candidate 1 should produce immediate perceived-speed gains without changing provider semantics. Candidate 2 is the best stability investment for DeepSeek V4, Xiaomi, Kimi K2.X, GLM 5.X, and Qwen 3.X because it turns provider quirks into a deep adapter module instead of scattered implementation checks.
